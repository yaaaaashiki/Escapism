1 はじめに
2000 年 Roy Thomas Fielding が REST (Repre-
sentational State Transfer) という Web アーキテク
チャスタイルを提唱した [1]．Web アプリケーション
は REST に沿って設計することで，サービス全体を
効率的に，可用性・信頼性高く開発できる．
近年，Twitter や LINE など，動的にリソースを
更新するアプリケーションが増加している．それに
伴い，クライアントとサーバ間の通信速度の高速化
やリソース更新の最適化に対する需要が高まってい
る．これらを実現するリアルタイム通信技術として，
WebSocket が開発された．
WebSocket は従来の擬似的なリアルタイム通信と
比べステートフルな通信により双方向通信を可能と
した．しかし，WebSocket は非リソース指向アーキ
テクチャである．本研究ではこの問題を解決するた
め REST と WebSocket アプリケーションを統合し
たサンプルアプリケーションを作成する．作成する
上で使用したライブラリを検証，比較する．またリ
アルタイムアプリケーションへの REST の追加につ
いて考察する．
2 基礎技術
2.1 REST
REST は，Roy Fielding [1] が Web の成功要因に
ついてアーキテクチャスタイルとしてまとめている．
Web サービスや Web API が REST の規約を守る
ことで Web 全体として調和が取れる．
2.2 WebSocket
WebSocket は TCP コネクションを使用した双方
向通信プロトコルである．従来の HTTP 通信と比
べ，通信コストが低い点やリアルタイム性の高い点
が評価されている [2] ．
Websocket-rails
2.2.1
Ruby on Rails 上で動作する WebSokcet ライブラ
リである．このライブラリはイベントと呼ばれる単
位でサーバとクライアント間でデータを送受信する．
2.2.2 Action Cable
Action cable は Ruby on Rails 5.0 において標準搭
載された．リアルタイム通信に対応したサーバ，クラ
イアントに対する JavaScript フレームワーク Ruby
フレームワークを同時に提供するフレームワークで
ある．従来サーバで全部レンダリングしていた部分
をページの一箇所を部分的にレンダリングできる．
3 提案手法
Action Cable 導入により，Rails アプリケーショ
ンの効率の良さとスケーラビリティを損なわずに，通
常の Rails アプリケーションと同じスタイル・方法
でリアルタイム機能を Ruby で記述できる．Ruby
on Rails には，DRY (Don’t Repeat yourself ), CoC
(Convention over Con(cid:12)guration) に基づいたコーデ
ィングできる設計に作られている．そこで Action
Cable における開発は Websocket-Rails と比べど
のようなメリットが得られるか検証する．
本研究では REST 設計におけるリアルタイムア
プリケーション用ライブラリを検証するため，機能
が同じサンプルアプリケーションを 2 つ作成した．
これに基づいてリアルタイムアプリケーションへの
REST の追加について考察する．

5.3 可読性
前節で述べた通り，Websocket-rails と比べ Ac-
tion Cable の方が CoC に基づいていることから可読
性の高いライブラリであるといえる．Action Cable
による開発は，コードの把握が容易となり開発者の
負担が減る.
6 リ ア ル タ イ ム ア プ リ ケ ー シ ョ ン へ の
REST の追加の検討
チャットアプリへのアドレス可能性の追加により
リソースに対しブラウザのブックマーク，戻る機能，
参照機能が回復した．また接続性の追加により，ユー
ザビリティが向上する．
WebSocket はステートフルな通信によってリアル
タイム通信を実現している．RESTful なアプリケー
ションの性質はステートレス性であるが，WebSocket
通信をステートレスにするとリアルタイム通信のメ
リットが消失する．そのため完全にステートレス性
にする必要はない．
7 まとめと今後の課題
本研究ではリアルタイムサンプルアプリケーショ
ンを作成し REST の性質を取り入れた．それに基づ
いてライブラリを検証，比較した．その結果，Action
Cable で開発することにより可読性の高いコーディン
グを可能にすることをを示した．また Action Cable
による開発のメリットについて考察した．さらにリ
アルタイムアプリケーションへの REST 追加につい
て考察した．今後の課題として，本研究で得られた
多くの知見を抽象化しそれをフレームワークとして
実装する．このフレームワークにより，Web と親和
性の高めるリアルタイムアプリケーション開発環境
を提供する.

5.1 DRY
Ruby on Rails で DRY に記述できるものは
Action Cable でも DRY に記述できる．つまり
Websocket-rails と Action Cable に違いが生じる
のは WebSocket 通信処理部分である．それらに関
して調査したが，Action Cable で DRY に記述でき
る処理は Websocket-rails でも DRY に記述でき
た．そのため本研究では，DRY に関して優位性を見
ることはできなかった．
5.2 CoC
Websocket-rails では，Websocket 通信の箇所を
JavaScript のメソッドで記述する．Ruby on Rails に
おいて JavaScript のコード記述位置は．開発者に依
存する．対して，Action Cable はコーディングの内
容と処理を記述するファイルが一意に定まる. Action
Cable は, 制約を設けることでシンプルかつロジッ
クに集中してコーディングできる．Action Cable は
Websocket-rails と比べ， CoC (Convention over
